# Token 消耗模型

> **版本标注**：本文档对应第一版预搜集方案。随分析方案演进（如引入两轮调用或 Agent 工具调用），需同步更新本文档。

## 概述

zentao-story-prd-analyzer 在分析阶段会将禅道条目和代码上下文组装为 prompt，发送给 LLM 进行判断。本文档记录各方案的 token 消耗构成和估算范围，便于成本评估和方案选型。

token 按 4 字符 ≈ 1 token 估算（`code_collector.py` 中 `TOKEN_ESTIMATE_RATIO = 4`）。

---

## 当前流程：关键词预搜集

### 数据流

```
禅道条目（标题/描述/元数据）
        │
        ▼
  关键词提取（ZentaoItem.keywords + --keywords + --symbols + --paths）
        │
        ▼
  本地代码搜索（rg → git grep → os.walk 降级）
        │
        ▼
  片段截断（max_total_tokens=8000）
        │
        ▼
  组装 prompt（模板指令 + 条目 + 代码上下文）
        │
        ▼
  单次 LLM 调用 → 解析 JSON 结果
```

### Token 构成

| 组成部分 | 估算 token | 说明 |
|---------|-----------|------|
| 模板指令 (`_FEATURE_TEMPLATE` / `_DEFECT_TEMPLATE`) | 400–500 | 固定开销，含 JSON Schema 约束 |
| 禅道条目（标题 + 描述 + 元数据） | 200–800 | 取决于条目描述长度 |
| 代码上下文 | 0–8000 | 受 `max_total_tokens` 上限控制 |
| **单次调用合计** | **600–9300** | 多数情况下代码量远低于上限 |

### 特点

- **优点**：单次 LLM 调用，延迟最低；token 成本可控且可预测。
- **缺点**：依赖关键词匹配质量，对 C/C++ 嵌入式项目关键词明确时效果好，对描述模糊或跨模块条目可能遗漏相关代码。
- **适用场景**：关键词明确的项目，成本敏感场景。

## 方案 A：两轮调用（LLM 辅助搜索）

### 数据流

```
第 1 轮：禅道条目 → LLM → 返回需搜索的关键词/路径/符号
        │
        ▼
  本地代码搜索（使用 LLM 返回的线索）
        │
        ▼
第 2 轮：条目 + 代码上下文 → LLM → 返回分析结果 JSON
```

### Token 构成

| 组成部分 | 估算 token | 说明 |
|---------|-----------|------|
| 第 1 轮 prompt（条目 + 搜索指令） | 800–1200 | 仅条目信息，不含代码 |
| 第 1 轮输出（关键词/路径列表） | 200–500 | LLM 返回搜索方向 |
| 第 2 轮 prompt（模板 + 条目 + 代码上下文） | 600–9300 | 与当前方案相同 |
| **两轮合计** | **1600–11000** | 比当前方案增加约 10% |

### 特点

- **优点**：搜索方向由 LLM 根据条目语义决定，比纯关键词匹配更精准。
- **缺点**：两次 LLM 调用，延迟翻倍；token 增加 10%；需实现第 1 轮的 prompt 模板和输出解析。
- **适用场景**：当预搜集结果为空（"未找到相关代码证据"）时的回退策略；关键词不明确的项目。

## 方案 B：Agent 工具调用（LLM 自主搜索）

### 数据流

```
禅道条目 → LLM (带 file_read 工具)
        │
        ▼ (多轮)
  LLM 决定调用 file_read(file_path) → 返回文件内容 → LLM 继续判断
        │
        ▼ (3–5 轮后)
  LLM 返回最终分析结果 JSON
```

### Token 构成

| 组成部分 | 估算 token | 说明 |
|---------|-----------|------|
| 初始 prompt（条目 + 工具定义） | 600–1000 | 含 tool schema |
| 每轮工具调用 + 返回 | 2000–4000 | 累积上下文，逐轮增长 |
| 3–5 轮合计 | **12000–24000** | 上下文累积效应显著 |

### 特点

- **优点**：LLM 自主决定要看的文件，精准度最高；无需离线搜索逻辑。
- **缺点**：token 成本增加 50%–200%；延迟 3–5 次 API 调用；实现复杂度高（需要 function calling / tool use 框架）；上下文累积可能导致截断。
- **适用场景**：需要高精度分析的通用场景；非 C/C++ 项目；复杂跨模块条目。

## 方案对比总表

| 维度 | 当前（预搜集） | 方案 A（两轮） | 方案 B（Agent 工具） |
|------|-------------|-------------|------------------|
| 准确性 | 中（依赖关键词） | 中高（LLM 辅助搜索） | 高（LLM 自主决策） |
| Token 成本 | 基线 ≤9K | 基线 +10% ≤11K | 基线 +50%–200% 12K–24K |
| 延迟 | 1 次 API 调用 | 2 次 API 调用 | 3–5 次 API 调用 |
| 实现复杂度 | 低 | 中 | 高 |
| 成本可预测性 | 高（上限固定） | 中（第 1 轮开销固定） | 低（取决于 LLM 决策） |
| 适用项目 | C/C++ 嵌入式，关键词明确 | 关键词模糊时的回退 | 通用场景，非 C/C++ 项目 |
